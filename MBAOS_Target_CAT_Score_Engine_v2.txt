MBAOS
Target CAT Score Engine
College-Specific Selection Logic, Probability Model & Back-Solved Percentile Architecture
Technical Design Document — Version 2.0
Module: Profile Evaluator → CAT Target Predictor → Shortlisting Engine
Prepared for engineering handoff
Why this document exists
The MVP PRD (v1.0) defines a single flat formula: Base College Cutoff + Profile Adjustment = Recommended CAT Target. That formula treats every college the same way, even though IIM Ahmedabad, IIM Bangalore, and SPJIMR each weigh CAT score, academics, work experience, and interview performance differently.
This document replaces that flat formula with a two-stage engine: (1) compute the student's admission probability at a specific college using that college's own weight table, then (2) algebraically back-solve the exact CAT percentile the student needs at that same college to cross a target probability threshold. The result: the same student gets a different required CAT score for IIM Ahmedabad vs IIM Bangalore vs SPJIMR — because the underlying math now actually uses each college's selection criteria, not just its cutoff number.

1. Problem Statement & Goal
The student's primary question is:
STUDENT INTENT
"I want to get into [College X]. Based on last year's cutoff and percentile data, and based on MY profile, what CAT score / percentile do I actually need?"
This is harder than a single lookup because colleges don't select students on CAT score alone. The Admission Criteria Database (PRD §10) already captures this — each college has its own exam_weight, academic_weight, workex_weight, and interview_weight — but the v1.0 CAT Target Predictor (PRD §11) never actually uses those weights. It only does cutoff + flat adjustment. This document fixes that gap.
1.1 Design Goals
	•	Use each college's actual selection weightage, not a one-size-fits-all cutoff adjustment.
	•	Produce a different required CAT percentile per college for the same student (this is the whole point — IIM Ahmedabad and IIM Bangalore must not return the same number just because the cutoffs are close).
	•	Be transparent about assumptions — anything not yet known about the student (e.g. interview performance, pre-interview) must be clearly flagged as an assumed default, not silently baked in.
	•	Stay computable in real time on form submit — no external calls, no ML training step required for MVP.
	•	Feed directly into the existing Shortlisting Engine (Dream / Target / Safe) using probability, not just percentile gap.

2.2 Why a Separate Target CAT Score Engine
The Profile Scoring Engine produces one number for the student (the Profile Score, 0–100) that is college-agnostic. The Target CAT Score Engine is college-aware — it is called once per college the student is comparing against, because the required CAT percentile genuinely differs by college. Separating these two responsibilities means:
	•	The Profile Score doesn't need to be recalculated when the student adds another college to compare.
	•	College weight changes (admission_criteria table) never require touching the Profile Scoring Engine.
	•	The engine can be called for one college (detail view) or batched across 10–15 colleges (shortlist view) without duplicating logic.

4. Stage 1 — Composite Score & Admission Probability
Goal: produce a single number representing how competitive the student's full profile is at a specific college, using that college's own weights — not a generic average.
4.1 Per-Component Normalization (0–100 scale)
Before applying college weights, each input must be normalized to a common 0–100 scale so weights are comparable regardless of each component's native scale.

4.2 Composite Score Formula
CompositeScore(college) =
      (CAT_percentile        × exam_weight)
    + (Academic_norm         × academic_weight)
    + (WorkEx_norm           × workex_weight)
    + (Interview_assumed     × interview_weight)
 
  where each weight is a fraction (e.g. 0.65, not 65) and weights sum to 1.0
  for that college. Result is on a 0–100 scale.
The worked examples below use one consistent student profile throughout this document — Academic Score normalized to 90.0, WorkEx Score normalized to 80.0, mock CAT percentile 97.0 — so the IIM Ahmedabad and SPJIMR results can be compared directly.

4.3 Handling the Interview Component (Pre-Interview Stage)
	•	Default assumption = 70/100 ("average performer"), applied uniformly so it doesn't arbitrarily help or hurt any one student.
	•	The Final Report MUST display this assumption explicitly: "Interview component assumed at average (70/100) — your actual target may shift once interview shortlist data is available."
	•	Once a student logs an actual interview/WAT-PI score (future module, not in MVP scope), the engine re-runs Stage 1 with the real value instead of the default.
4.4 Colleges With No Interview Weight (e.g. IIM Ahmedabad)
Per the Admission Criteria Database (PRD §10), IIM Ahmedabad's listed weights are CAT 65%, Academics 25%, WorkEx 10% — summing to 100% with no interview component recorded. Rule:
	•	If a college's stored weights already sum to 100%, use them as-is. Do not inject an interview term.
	•	If a college's stored weights sum to less than 100% (data entry gap) and no interview_weight is set, the missing percentage is redistributed proportionally across the existing components — not silently dropped, and not assumed as interview score. This is a data-quality safeguard, not a scoring rule. Flag any college with this gap in an admin data-quality report.
4.5 Stage 1 Output — Admission Probability
Composite Score alone isn't meaningful across colleges (an 85 at IMI does not mean the same admit chance as an 85 at IIM Ahmedabad). Convert to probability using a logistic curve calibrated per college tier:
P(admit) = 1 / (1 + e^(-k × (CompositeScore − T)))
 
  T = college's calibrated composite threshold
      (derived once per college by converting its historical cutoff_percentile
       into composite-score-equivalent terms — stored in admission_criteria as
       'calibrated_threshold', recomputed yearly when new cutoff data lands)
 
  k = steepness constant — controls how sharply probability rises near the threshold
      Default k = 0.15 for MVP (single global constant, tunable per college tier later)

5. Stage 2 — Back-Solving the Required CAT Percentile
Goal: instead of just reporting today's probability, tell the student exactly what CAT percentile they need to reach a target probability — typically 80% for a college to be realistically classified as "Target" rather than "Dream".
5.1 Inverting the Logistic Function
Because Composite Score is linear in CAT percentile (it's just one weighted term among several), the logistic relationship can be algebraically inverted to solve directly for the CAT percentile that produces any desired probability — no trial-and-error, no simulation loop.
Step 1 — invert probability to required Composite Score:
 
    CompositeScore_required = T + ln( P / (1 − P) ) / k
 
    where P = target probability (0.80 default for 'Target' band, 0.50 for 'Dream' band)
 
Step 2 — isolate the CAT percentile term from the weighted sum:
 
    CompositeScore_required =
          (CAT_required      × exam_weight)
        + (Academic_norm     × academic_weight)        ← already known, fixed
        + (WorkEx_norm       × workex_weight)           ← already known, fixed
        + (Interview_assumed × interview_weight)        ← already known, fixed
 
    ⇒  CAT_required =
         ( CompositeScore_required
           − Academic_norm×academic_weight
           − WorkEx_norm×workex_weight
           − Interview_assumed×interview_weight )
         ÷ exam_weight
 
Step 3 — clamp to valid percentile bounds:
    CAT_required = MIN( MAX(CAT_required, 50), 99.9 )

5.3 Converting Required Percentile → Required Score → Expected Rank
PRD §8's cat_percentile_mapping table is sparse (only ~6 reference rows: 90, 95, 98, 99, 99.5, 99.9). A required percentile of 98.5 or 97.3 will not have an exact row. Rule:
	•	Linear interpolation between the two nearest stored percentile rows for both raw_score and approx_rank.
	•	Example: required percentile 98.5 sits between stored rows 98 (score 85, rank 6000) and 99 (score 95, rank 3000). Interpolated score = 85 + (98.5−98)/(99−98)×(95−85) = 90. Interpolated rank = 6000 + (98.5−98)/(99−98)×(3000−6000) = 4500.
	•	If required percentile exceeds the highest stored row (99.9), do not extrapolate beyond it — display the 99.9 row's score/rank as a ceiling value with a label "99.9+".
	•	Recompute interpolation inputs yearly when new cat_percentile_mapping rows are added (PRD §8 requires minimum 5 years stored).
6. Shortlisting Engine — Probability-Driven Classification
PRD §12 classifies colleges by percentile gap OR probability band. With Stage 1 now producing a real probability per college, classification becomes consistent and removes the ambiguity of the original "gap > 3 OR probability < 50%" dual condition.

8. Edge Cases & Validation Rules
Edge Case
Rule
College weights in DB don't sum to 1.000
Reject on data entry / flag in admin panel — never silently normalize at query time, since that masks data-entry errors (see PRD bug review).
Back-solved CAT percentile > 99.9
Clamp to 99.9 and explicitly label the college as mathematically capped below target probability, per §5.2.
Back-solved CAT percentile < 50
Clamp to 50 floor — a college this far below the student's profile should already classify as Safe; do not display percentiles that look like "go easier than the exam's lower bound".
Student has no mock percentile yet
Stage 1 uses category-default Exam Readiness score (10, per PRD §5) converted to an assumed percentile equivalent (~75th, the midpoint of the 'no mock' default band) — flagged in report as 'Estimated — take a mock for an accurate number.'
Required percentile needed exceeds highest stored cat_percentile_mapping row
Do not extrapolate past 99.9; show as a ceiling value (§5.3).
College has interview_weight = 0 (e.g. IIM Ahmedabad)
Interview_assumed term simply contributes 0 — no default of 70 is invented for colleges that don't use an interview stage.
